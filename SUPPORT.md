# ArcherDB Support

Use the right channel so maintainers can respond efficiently and security issues
stay private when necessary.

## Where To Go

- Bugs: open a GitHub issue with reproduction steps, expected behavior, actual
  behavior, and environment details.
- Feature requests: open a GitHub issue describing the use case, constraints,
  and proposed behavior.
- Documentation fixes: open a GitHub issue or send a pull request directly for
  small corrections.
- Security vulnerabilities: follow [SECURITY.md](SECURITY.md). Do not use public
  issues for security reports.

## Before Opening An Issue

- Read [README.md](README.md) and the relevant content under [docs/](docs/README.md).
- Search existing issues to avoid duplicates.
- For behavior questions, include the exact command, SDK call, or config that
  produced the result you are asking about.

## What Maintainers Need

The fastest issue reports usually include:

- ArcherDB version or commit
- operating system and architecture
- exact command or client code
- logs, stack traces, or error messages
- steps to reproduce
- whether the problem reproduces on a clean checkout

## Response Expectations

ArcherDB is maintained in the open, but response times are best-effort. Issues
with clear reproductions and narrow scope are easier to triage quickly.

If your report is actually a vulnerability, stop and use the private reporting
path in [SECURITY.md](SECURITY.md).

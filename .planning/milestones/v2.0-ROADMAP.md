# Milestone v2.0: Performance & Scale

**Status:** SHIPPED 2026-01-26
**Phases:** 11-18
**Total Plans:** 53

## Overview

ArcherDB v2.0 transforms the production-ready v1.0 into an enterprise-scale database. The roadmap follows a measurement-first philosophy: build comprehensive profiling infrastructure before any optimization work, then proceed through storage, memory, query, consensus, and scale-out optimizations in dependency order.

## Milestone Summary

**Key Decisions:**

- Measurement-first approach: All optimization requires profiling data
- Phase order follows dependency/risk: low-risk measurement -> high-risk consensus/sharding
- Breaking changes grouped for coordinated v2.0 release
- Hot shard scoring normalized to load_shedding thresholds
- Default fan-out policy: all for uuid_batch, majority for radius/polygon/latest
- Exit code 2 for flag conflicts in benchmarks (distinct from general error 1)
- Hybrid compaction benchmark: actual ArcherDB for tiered, scaled estimate for leveled
- Query metrics instances declared at module level (pub var) for thread-safe access

**Issues Resolved:**

- Compression benchmark negative ratios (fixed with datafile delta measurement)
- Compaction benchmark throughput too low (fixed with hybrid approach)
- Require-archerdb flag not blocking dry-run (fixed with conflict check)
- Query metrics not exported (wired into Registry.format())
- Load shedding HTTP 429 not wired (added to metrics_server)

**Issues Deferred:**

- None - all v2.0 requirements satisfied

**Technical Debt Incurred:**

- ~90 TODOs remain in infrastructure code (Zig language limitations, pre-existing)
- Pre-existing flaky tests in ram_index.zig (concurrent/resize stress tests)

# Phase 3: Core Geospatial - Research

**Researched:** 2026-01-22
**Domain:** Geospatial indexing verification (S2, queries, entity ops, RAM index)
**Confidence:** HIGH

## Summary

This research investigates verification strategies for ArcherDB's existing geospatial functionality. The phase is about **verification of correctness**, not new feature development. ArcherDB already has a complete S2 implementation in pure Zig (`src/s2/`), query engines for radius and polygon operations (`src/s2_index.zig`, `src/post_filter.zig`), entity operations in the GeoStateMachine (`src/geo_state_machine.zig`), and a RAM index (`src/ram_index.zig`).

Key findings:
- The S2 implementation uses deterministic software trigonometry (Chebyshev/CORDIC) to ensure bit-exact results across all platforms - critical for VSR consensus
- Golden vector validation against Google S2 reference is already partially implemented but needs comprehensive expansion
- The RAM index already has the race condition fix at line 1859 (`remove_if_id_matches`) for TTL expiration
- Polygon validation already handles CCW winding, self-intersection detection, and hole containment

**Primary recommendation:** Create exhaustive test vector sets comparing ArcherDB's S2 implementation against Google's reference, then verify all query types return correct results through property-based testing.


### Pattern 2: Determinism Verification via Hash
**What:** XOR hash of many operations must match known constant
**When to use:** Cross-platform determinism testing
**Example:**
```zig
// Source: src/s2/cell_id.zig - existing pattern
test "cross-platform determinism" {
    var hash: u64 = 0;
    // Grid of coordinates, all levels, all operations
    for (lat_steps) |lat| {
        for (lon_steps) |lon| {
            for (levels) |lvl| {
                const cell_id = fromLatLonNano(lat, lon, lvl);
                hash ^= cell_id;
                // Also test hierarchy, round-trip
            }
        }
    }
    // Expected hash computed on Linux x86_64
    const expected_hash: u64 = 0xcfdb4dbdd12dfa59;
    try std.testing.expectEqual(expected_hash, hash);
}
```

### Pattern 3: Property-Based Query Testing
**What:** For radius queries: all points returned are within radius, none outside
**When to use:** Query correctness verification
**Example:**
```zig
// Verify: returned points are within radius
for (results) |event| {
    const dist = S2.distance(center_lat, center_lon, event.lat_nano, event.lon_nano);
    try std.testing.expect(dist <= radius_mm);
}

// Verify: no qualifying points were missed (sample check)
for (all_events) |event| {
    if (S2.isWithinDistance(center_lat, center_lon, event.lat_nano, event.lon_nano, radius_mm)) {
        // Must be in results
        try std.testing.expect(containsEntity(results, event.entity_id));
    }
}
```

### Anti-Patterns to Avoid
- **Floating-point comparison without tolerance**: Use `expectApproxEqAbs` for float comparisons
- **Testing against std.math trig**: These are non-deterministic; use the software trig in `src/s2/math.zig`
- **Relying on hardware FPU**: The deterministic math avoids FSIN/FCOS instructions deliberately

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Distance calculation | Manual haversine | `S2.distance()` in s2_index.zig | Already deterministic, tested |
| Point-in-polygon | Custom ray casting | `S2.pointInPolygon()` | Handles edge cases, holes |
| Cell ID computation | Custom projection | `s2.latLonToCellId()` | Matches Google S2 exactly |
| S2 cell covering | Manual recursion | `RegionCoverer.coverCap()` | Optimized, handles edge cases |
| Winding order check | Manual area calc | `S2.isCounterClockwise()` | Uses shoelace formula correctly |

**Key insight:** The S2 implementation is complete - verification is about proving correctness, not reimplementing.

## Common Pitfalls

### Pitfall 1: Non-Deterministic Trigonometry
**What goes wrong:** Different platforms produce different S2 cell IDs for same coordinates
**Why it happens:** Hardware FSIN/FCOS instructions vary by CPU
**How to avoid:** All trig goes through `src/s2/math.zig` (already implemented)
**Warning signs:** Test hash mismatch on ARM vs x86

### Pitfall 2: Antimeridian Edge Cases
**What goes wrong:** Polygons crossing 180 longitude fail or return wrong results
**Why it happens:** Naive bounding box checks fail when min_lon > max_lon
**How to avoid:** Test vectors must include antimeridian cases explicitly
**Warning signs:** Alaska/Russia/Fiji queries return unexpected results

### Pitfall 3: Polar Region Coordinate Singularities
**What goes wrong:** Longitude becomes meaningless near poles
**Why it happens:** Longitude lines converge at poles
**How to avoid:** Test vectors at lat=89.999, lat=-89.999
**Warning signs:** S2 face assignment wrong near poles

### Pitfall 4: Cell Boundary Inclusivity
**What goes wrong:** Points exactly on cell boundaries counted twice or zero times
**Why it happens:** Floating-point rounding at boundaries
**How to avoid:** Test exact boundary coordinates
**Warning signs:** Query results differ by 1 at cell boundaries

### Pitfall 5: Race Condition in TTL Expiration (Line 1859)
**What goes wrong:** Freshly inserted data deleted by concurrent TTL cleanup
**Why it happens:** Check and remove not atomic
**How to avoid:** Use `remove_if_id_matches()` which checks latest_id (already implemented)
**Warning signs:** `race_detected: true` in RemoveIfMatchResult

### Pitfall 6: Tombstone Accumulation in RAM Index
**What goes wrong:** Index performance degrades over time
**Why it happens:** Deleted entities leave tombstones that aren't reclaimed
**How to avoid:** Monitor `stats.tombstone_ratio()`, rebuild when > 30%
**Warning signs:** `max_probe_length_seen` growing, latency increasing

## Open Questions

Things that couldn't be fully resolved:

1. **WGS84 vs Sphere Assumptions**
   - What we know: S2 uses unit sphere, distances use Earth mean radius (6371008.8m)
   - What's unclear: User mentioned "WGS84 vs sphere assumptions explicitly documented and tested"
   - Recommendation: Document in verification tests that S2 uses spherical model, not WGS84 ellipsoid

2. **Stress Testing vs Formal Analysis for Race Condition**
   - What we know: The race condition at line 1859 is already fixed with `remove_if_id_matches()`
   - What's unclear: Which verification method to use (per "Claude's Discretion" in CONTEXT.md)
   - Recommendation: Use stress testing - concurrent upsert and TTL expiration threads, verify no data loss

3. **Batch Operation Atomicity**
   - What we know: VSR commit is atomic, individual operations may not be
   - What's unclear: Whether batch insert/delete should be all-or-nothing
   - Recommendation: Document current behavior, verify with VSR transaction model

## Verification Strategy Summary

### S2 Indexing (S2-01 through S2-08)
1. Generate exhaustive test vectors from Google S2 (Go)
2. Run golden vector validation test (already stubbed in s2.zig)
3. Verify cross-platform determinism hash
4. Benchmark S2 cell computation performance
5. Document memory usage per cell level

### Radius Queries (RAD-01 through RAD-08)
1. Property test: all results within radius, none outside
2. Edge cases: zero radius, huge radius (whole Earth)
3. Verify Haversine (great-circle) distance
4. Test high-density clusters (many points in small area)
5. Verify deterministic ordering of results

### Polygon Queries (POLY-01 through POLY-09)
1. Test convex polygons (triangles, squares)
2. Test concave polygons (L-shapes, stars)
3. Test polygons with holes (donuts)
4. Verify self-intersecting polygon rejection
5. Test antimeridian-crossing polygons (Alaska, Fiji)
6. Benchmark complex polygons (10,000+ vertices)

### Entity Operations (ENT-01 through ENT-10)
1. Insert/upsert/delete round-trip verification
2. Tombstone creation and propagation
3. GDPR deletion verification (entity not retrievable)
4. UUID and batch UUID query correctness
5. TTL expiration with metrics

### RAM Index (RAM-01 through RAM-08)
1. O(1) lookup verification under load
2. Concurrent access correctness (VSR single-writer assumption)
3. Race condition stress test (line 1859)
4. Memory usage bounds (64 bytes * capacity)
5. Checkpoint/restart recovery
6. Hash collision handling
7. TTL integration with `remove_if_id_matches()`

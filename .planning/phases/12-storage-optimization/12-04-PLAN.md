<objective>
Implement latency-driven compaction throttling to prevent I/O spikes from impacting query latency.

Purpose: During heavy compaction, I/O contention can cause query latency spikes. This plan adds a throttle that reduces compaction throughput when P99 query latency exceeds a configurable threshold. The throttle responds quickly to latency spikes but recovers gradually to prevent oscillation.

Output: ThrottleState with latency-driven throughput adjustment, configuration options, integration with compaction resource pool.
</objective>

  <action>
  Create src/lsm/compaction_throttle.zig with latency-driven throttling:

  1. Define ThrottleConfig (TiKV-style predictive + reactive fallback):
     ```zig
     pub const ThrottleConfig = struct {
         // PREDICTIVE thresholds (primary - TiKV pattern)
         // Check pending compaction bytes FIRST to prevent stalls before they happen
         /// Start slowing when pending bytes exceed this (64 GiB default)
         soft_pending_compaction_bytes: u64 = 64 * 1024 * 1024 * 1024,
         /// Aggressive slowdown above this (256 GiB default)
         hard_pending_compaction_bytes: u64 = 256 * 1024 * 1024 * 1024,

         // REACTIVE fallback (secondary - when pending bytes unavailable or low)
         /// Start throttling when P99 exceeds this (milliseconds)
         p99_latency_threshold_ms: f64 = 50.0,
         /// Emergency throttle above this (milliseconds)
         p99_latency_critical_ms: f64 = 100.0,

         /// How often to check (milliseconds)
         check_interval_ms: u64 = 1000,
         /// Throughput adjustment step per check (10% = 0.1)
         throttle_ratio_step: f64 = 0.1,
         /// Never go below this throughput (10% = 0.1)
         min_throughput_ratio: f64 = 0.1,
         /// Hysteresis: latency must drop this much below threshold to recover
         recovery_hysteresis_ms: f64 = 10.0,
         /// Require this many consecutive good checks before recovery
         consecutive_good_checks_required: u32 = 3,
     };
     ```

  2. Define ThrottleState and ThrottleMetrics:
     ```zig
     pub const ThrottleMetrics = struct {
         pending_compaction_bytes: u64,  // Primary signal
         current_p99_ms: f64,             // Fallback signal
     };

     pub const ThrottleState = struct {
         current_throughput_ratio: f64 = 1.0,  // 1.0 = full speed
         last_check_ns: i128 = 0,
         consecutive_good_checks: u32 = 0,
         in_critical: bool = false,

         pub fn init() ThrottleState { return .{}; }
     };
     ```

  3. Implement update() method with PREDICTIVE + REACTIVE logic (TiKV pattern):
     ```zig
     pub fn update(self: *ThrottleState, metrics: ThrottleMetrics, config: ThrottleConfig) void
     ```
     Logic (check in order - predictive FIRST):
     - PREDICTIVE PATH (pending bytes - prevents stalls before they happen):
       a. If pending_bytes > hard_pending: halve throughput immediately (aggressive)
       b. Else if pending_bytes > soft_pending: decrease ratio by step
       c. If either predictive triggered: reset good checks and return
     - REACTIVE FALLBACK (P99 latency - when pending bytes low):
       a. If current_p99_ms > critical: immediately drop to min_throughput_ratio
       b. Else if current_p99_ms > threshold: decrease ratio by step
       c. Reset consecutive_good_checks on any bad check
     - RECOVERY (when both signals are good):
       a. If pending_bytes < soft AND p99 < threshold - hysteresis:
          - Increment consecutive_good_checks
          - If >= required: increase ratio by step (clamp to 1.0)

  4. Implement should_check() method:
     - Return true if enough time has passed since last_check_ns
     - Uses std.time.nanoTimestamp()

  5. Implement get_delay_ns() method:
     - Given desired work duration, return how long to sleep
     - delay = work_duration * (1/throughput_ratio - 1)
     - If ratio == 1.0, return 0 (no delay)

  6. Add unit tests:
     - Test predictive path: hard pending bytes triggers aggressive slowdown
     - Test predictive path: soft pending bytes triggers gradual slowdown
     - Test reactive fallback: P99 critical triggers immediate min
     - Test reactive fallback: P99 threshold triggers gradual slowdown
     - Test hysteresis prevents oscillation
     - Test recovery after consecutive good checks (both signals good)

  Follow TiKV-style predictive throttling patterns from 12-RESEARCH.md.

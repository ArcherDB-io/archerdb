  <name>Task 1: Implement read path optimizations based on bottleneck analysis</name>
  <files>src/archerdb/</files>
  <action>
Based on the bottleneck analysis in 05-01-SUMMARY.md, implement targeted read path optimizations.

**Common optimization techniques (apply based on flame graph findings):**

1. **Cache tuning** - If flame graph shows grid reads hot:
   - Increase `cache_geo_events_size_default` for geo event cache
   - Increase `grid_cache_size_default` for block cache
   - Review cache eviction policy (LRU effectiveness)

2. **Index optimization** - If RAM index lookups are hot:
   - Optimize hash table probe sequence
   - Consider index structure improvements
   - Review S2 cell ID computation caching

3. **Spatial query optimization** - If S2 operations are hot:
   - Cache S2 cell coverings for repeated queries
   - Optimize region coverer parameters (max_cells, min/max_level)
   - Batch cell lookups where possible

4. **I/O path optimization** - If grid block reads are hot:
   - Tune read-ahead for sequential scans
   - Optimize block layout for query patterns
   - Review io_uring completion batching

5. **Result assembly optimization** - If result building is hot:
   - Pre-allocate result buffers
   - Reduce allocations in result iteration
   - Optimize serialization path

**Implementation approach:**
- Focus on optimizations that affect latency (not throughput)
- Make ONE optimization at a time
- Measure P99 latency impact after each change
- Keep changes that show >= 10% P99 improvement

**CRITICAL:** Run unit tests after each change:
```bash
./zig/zig build -j4 -Dconfig=lite test:unit
```

Per RESEARCH.md Pitfall 5: Watch for cache misses in spatial queries. High cache-misses in perf stat indicates S2 cell coverings touching non-contiguous memory.

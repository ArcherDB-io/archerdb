  <name>Task 1: Implement write path optimizations based on bottleneck analysis</name>
  <files>src/archerdb/</files>
  <action>
Based on the bottleneck analysis in 05-01-SUMMARY.md, implement targeted write path optimizations.

**Common optimization techniques (apply based on flame graph findings):**

1. **Batching optimization** - If flame graph shows syscall overhead:
   - Increase batch coalescing in message handling
   - Reduce number of io_uring submissions per logical write
   - Tune `message_size_max` and batch accumulation logic

2. **Memory allocation optimization** - If flame graph shows allocator hot:
   - Use arena allocators for request-scoped allocations
   - Pre-allocate buffers for common paths
   - Reduce temporary allocations in hot loops

3. **Compaction tuning** - If flame graph shows compaction contention:
   - Adjust `compaction_throttle_*` settings in config
   - Tune `lsm_growth_factor` for write-heavy workload
   - Consider tiered compaction parameters

4. **io_uring tuning** - If I/O path is bottleneck:
   - Increase `grid_iops_write_max` if queue depth limited
   - Check for SQE submission batching opportunities
   - Verify completion handling efficiency

5. **Index update optimization** - If RAM index updates are hot:
   - Consider batch index updates
   - Optimize hash table operations
   - Review S2 cell computation caching

**Implementation approach:**
- Make ONE optimization at a time
- Measure impact after each change
- Keep changes that show >= 5% improvement
- Revert changes that don't measurably help

**CRITICAL:** Run unit tests after each change:
```bash
./zig/zig build -j4 -Dconfig=lite test:unit
```

Per RESEARCH.md Pitfall 1: Lite config has throughput ceiling due to message_size_max=32KB. Optimizations should still be measurable but final validation needs production config.

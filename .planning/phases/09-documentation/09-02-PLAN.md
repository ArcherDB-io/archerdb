Append the second half of docs/architecture.md covering geospatial indexing and distributed features:

**5. S2 Geospatial Indexing** (ARCH-03)
- What S2 is: Google's spherical geometry library, Hilbert curve mapping
- How S2 cells work: hierarchical decomposition, 30 levels, cell IDs
- Why S2: locality preservation, efficient covering algorithms
- Query flow:
  - Radius query: compute covering cells, scan cells, filter by distance
  - Polygon query: compute covering cells, scan cells, point-in-polygon test
- Performance characteristics: O(n) where n = entities in covering cells
- Mermaid diagram showing S2 cell hierarchy
- Key insight: Why S2 over geohash or R-tree

**6. RAM Index** (ARCH-04)
- What RAM index provides: O(1) latest position lookup per entity
- Design: hash table mapping entity_id -> latest GeoEvent
- Memory footprint: ~64 bytes per entity (formula from 03-05 decision)
- Persistence: reconstructed from LSM on restart, checkpointed
- Concurrency: lock-free reads, mutex-protected writes
- Key insight: Why a separate RAM index vs querying LSM

**7. Sharding** (ARCH-05)
- Why shard: scale beyond single node, geographic distribution
- Sharding strategies:
  - Jump hash (default): uniform distribution, minimal resharding movement
  - Modulo: legacy, power-of-2 only
- Shard routing: client computes shard from entity_id
- Cross-shard queries: coordinator fans out, aggregates results
- Mermaid diagram showing sharded architecture

**8. Replication** (ARCH-06)
- Two modes: synchronous (VSR within region), asynchronous (cross-region)
- Cross-region flow:
  - Primary commits locally via VSR
  - WAL entries shipped to S3 (or disk spillover if S3 unavailable)
  - Followers consume from S3, apply to local state
- Consistency model: strong within region, eventual across regions
- Failure handling: disk spillover during S3 outages
- Mermaid diagram showing cross-region replication flow

**9. Summary**
- Design philosophy recap
- Trade-offs made and why
- Link to operations docs for running in production

Ensure each section explains WHY design choices were made, not just what they are (per CONTEXT.md specifics).

**MULTI-07: Cluster reconfiguration (add/remove nodes)**

The test framework primarily supports testing replica replacement via reformat (replacing a failed node with a fresh one). For true dynamic reconfiguration, the membership module would need to be exercised.

Based on replica_test.zig patterns:
```zig
test "MULTI-07: replace failed replica via reformat" {
    // Test that a completely failed replica can be replaced
    // This simulates the add/remove workflow: remove bad node, add fresh one

    const t = try TestContext.init(.{ .replica_count = 3 });
    defer t.deinit();

    var c = t.clients(.{});
    try c.request(5, 5);

    var b2 = t.replica(.B2);

    // Simulate catastrophic failure (corrupt everything)
    b2.stop();

    // Cluster continues with 2/3
    try c.request(10, 10);
    try expectEqual(t.replica(.A0).commit(), 10);
    try expectEqual(t.replica(.B1).commit(), 10);

    // Replace the failed replica via reformat (simulates adding replacement node)
    try b2.open_reformat();
    t.run();

    // Verify the new replica joined and synced
    try expectEqual(b2.status(), .normal);
    try expectEqual(b2.commit(), 10);

    // Verify cluster is fully operational
    try c.request(15, 15);
    try expectEqual(t.replica(.R_).commit(), 15);
}
```

Note: This tests replica replacement (the practical reconfiguration scenario). Dynamic membership changes (changing N itself) would require additional framework support not commonly tested in replica_test.zig.

If the test reveals that open_reformat() is not appropriate for this scenario, use the pattern from "Cluster: recovery: reformat unrecoverable replica" test in replica_test.zig as reference.

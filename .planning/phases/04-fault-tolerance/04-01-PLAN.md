For FAULT-01 (Process crash survives without data loss):

1. Create test "FAULT-01: process crash (SIGKILL) survives without data loss (R=3)"
   - Create 3-node cluster, commit 10+ operations
   - Stop one replica abruptly (simulating SIGKILL via stop())
   - Restart the replica
   - Verify: replica recovers to correct commit position
   - Verify: cluster can continue accepting new operations
   - Verify: all replicas converge to same state

2. Create test "FAULT-01: process crash during pending writes (R=3)"
   - Create cluster, commit some operations
   - Stop replica DURING operations (stop while requests in flight)
   - Restart replica
   - Verify: no data loss for committed operations
   - Verify: uncommitted operations handled correctly (client retry)

3. Create test "FAULT-01: multiple sequential crashes (R=3)"
   - Crash and restart each replica in sequence
   - Between each crash/restart cycle, commit more operations
   - Verify: no cumulative data loss across multiple crashes
   - Verify: final state is consistent across all replicas

Implementation note: stop() simulates SIGKILL by immediately terminating the replica without cleanup.

  <action>
Add tests for FAULT-02 and FAULT-07 to the same file.

For FAULT-02 (Power loss survives without data loss):

Power loss is simulated via torn writes - partial writes where the storage.reset() injects corruption on pending writes based on crash_fault_probability.

1. Create test "FAULT-02: power loss (torn writes) survives without data loss (R=3)"
   - Enable storage fault injection for torn writes
   - Commit operations
   - Stop all replicas (simulating power loss - pending writes may be torn)
   - Use corrupt() to simulate torn write on WAL prepare
   - Restart all replicas
   - Verify: cluster detects torn writes and repairs from intact replicas
   - Verify: committed data is preserved

2. Create test "FAULT-02: power loss during checkpoint (R=3)"
   - Commit enough operations to trigger checkpoint
   - Stop replicas during/after checkpoint
   - Corrupt checkpoint-related grid blocks (simulating torn checkpoint write)
   - Restart and verify: cluster recovers from disjoint corruption pattern
   - Verify: checkpoint data preserved through cross-replica repair

For FAULT-07 (Corrupted log entries recovered):

Per CONTEXT.md decision: "Corrupted data handling: Fail startup with clear error - require operator intervention rather than risk serving bad data"

1. Create test "FAULT-07: corrupted log entry detected via checksum (R=3)"
   - Commit operations
   - Stop one replica
   - Corrupt a WAL entry (not header - the actual prepare data)
   - Restart replica
   - Verify: corruption detected (enters recovering_head or similar)
   - Verify: cluster repairs from healthy replicas

2. Create test "FAULT-07: corrupted log entry on single replica (R=1) - clear error"
   - Create R=1 cluster
   - Commit operations
   - Stop replica
   - Corrupt WAL entry between checkpoint and head
   - Attempt restart
   - Verify: returns error.WALCorrupt (clear error, not silent corruption)

3. Create test "FAULT-07: multiple corrupted entries across replicas recoverable"
   - Corrupt different entries on different replicas (disjoint pattern)
   - Verify: cluster can recover as long as each entry intact on at least one replica

Add FAULT-08 tests to the existing fault_tolerance_test.zig file.

For FAULT-08 (Recovery time < 60 seconds after crash):

Per CONTEXT.md: "Recovery time target: Under 60 seconds for replica to rejoin cluster after crash"

The recovery infrastructure tracks:
- recovery_start_ns: timestamp when recovery begins
- recovery_metrics: records recovery path and duration
- RecoveryPath: enum of none/wal_replay/lsm_scan/full_rebuild

1. Create test "FAULT-08: recovery from crash completes within 60 seconds (R=3)"
   - Create cluster, commit operations past checkpoint
   - Stop one replica (crash)
   - Record timestamp before restart
   - Restart replica
   - Wait for status to become .normal (recovery complete)
   - Calculate elapsed time
   - Verify: elapsed time < 60 seconds
   - Verify: replica is in normal status with correct commit

2. Create test "FAULT-08: recovery from WAL corruption within 60 seconds (R=3)"
   - Create cluster, commit operations
   - Stop one replica
   - Corrupt WAL prepare (requiring repair)
   - Record timestamp, restart replica
   - Wait for recovery and repair to complete
   - Verify: total time < 60 seconds

3. Create test "FAULT-08: recovery from grid corruption within 60 seconds (R=3)"
   - Create cluster, checkpoint to populate grid
   - Stop one replica
   - Corrupt grid blocks (requiring repair from cluster)
   - Record timestamp, restart replica
   - Wait for recovery and repair to complete
   - Verify: total time < 60 seconds

4. Create test "FAULT-08: recovery path classification correct"
   - Test different recovery scenarios
   - Verify: recovery_path is correctly classified (wal_replay, lsm_scan, etc.)
   - This validates the RecoveryPath enum is being used correctly

Implementation notes:

For timing in deterministic tests, use tick counts rather than wall clock time. The TestContext.run() method runs up to 4,100 ticks. Each tick represents simulated time.

Alternatively, if the replica exposes recovery_metrics, inspect those directly after recovery completes. The recovery_metrics struct tracks path and duration.

The test infrastructure runs in simulated time, so "60 seconds" translates to a tick count based on the tick rate. If the default tick rate is 1ms, then 60 seconds = 60,000 ticks.

If direct timing measurement isn't practical in the test infrastructure, verify that:
1. Recovery completes within a reasonable tick count
2. The recovery_metrics show path and duration
3. Status transitions to .normal

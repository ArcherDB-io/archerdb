  <name>Task 2: Compute hot shard score and update rebalance state</name>
  <files>src/archerdb/metrics_server.zig</files>
  <action>
    Extend handleHealthShards with a hot-shard scoring helper that derives a mixed
    signal per shard:
    - Throughput score: (read_rate + write_rate) / average_throughput
    - Queue score: cluster_metrics.archerdb_shed_queue_depth.get() / max_queue_depth
    - Latency score: metrics.Registry.read_latency.getExtendedStats().p99 / threshold_ms
    Use weights 0.34/0.33/0.33 (throughput/latency/queue), clamp each score to [0,1],
    and compute a 0-100 score. Select the hottest shard (highest score).

    Track cooldown + active migration slots with static state (last_rebalance_ns,
    active_moves) and config constants: threshold=0.70, ratio_guard=1.5, cooldown=300s,
    max_concurrent_moves=2. When score exceeds threshold and cooldown has elapsed,
    set shard_rebalance_needed to 1 and update cooldown_seconds accordingly.

    Update metrics.Registry shard_hot_id/shard_hot_score/shard_rebalance_* gauges on
    each /health/shards request so operators see current detection state.
  </action>
  <verify>./zig/zig build -j4 -Dconfig=lite test:unit -- --test-filter "metrics_server"</verify>
  <done>Hot shard detection updates metrics and respects cooldown/max concurrent limits.</done>

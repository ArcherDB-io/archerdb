   ```zig
   //! Upgrade Orchestration Module
   //!
   //! Provides safe rolling upgrade functionality with health-based rollback.
   //! Per CONTEXT.md: Upgrade followers first, then primary.
   //! Rollback triggers: health probe failures OR P99 doubles OR error rate threshold.

   const std = @import("std");
   const log = std.log.scoped(.upgrade);

   pub const UpgradeOptions = struct {
       /// Cluster addresses (comma-separated)
       addresses: []const u8,
       /// Target version to upgrade to
       target_version: []const u8,
       /// Dry run - show what would happen without executing
       dry_run: bool = false,
       /// Skip backup verification (not recommended)
       skip_backup_check: bool = false,
       /// Pause on failure instead of rolling back
       pause_on_failure: bool = true,
       /// P99 latency threshold in ms (rollback if exceeded)
       p99_threshold_ms: u64 = 200,
       /// Error rate threshold (rollback if exceeded)
       error_rate_threshold: f64 = 0.01,
       /// Health check interval between nodes
       health_check_interval_ms: u64 = 10_000,
       /// Number of health check retries before failure
       health_check_retries: u32 = 3,
       /// Timeout for catch-up after restart (ms)
       catchup_timeout_ms: u64 = 300_000, // 5 minutes
   };

   pub const UpgradeState = enum {
       not_started,
       preflight_checks,
       upgrading_followers,
       upgrading_primary,
       verifying,
       complete,
       paused,
       rolled_back,
       failed,
   };

   pub const ReplicaInfo = struct {
       address: []const u8,
       replica_id: u8,
       is_primary: bool,
       version: []const u8,
       healthy: bool,
   };
   ```

   Implement core functions:
   - `pub fn runPreflightChecks(self: *Upgrader) !void`
     - Verify backup exists and is recent
     - Check all replicas healthy
     - Identify primary vs followers
     - Verify target version compatibility
   - `pub fn identifyPrimary(self: *Upgrader) !?ReplicaInfo`
     - Query /metrics for archerdb_replica_role or /health/detailed
   - `pub fn upgradeReplica(self: *Upgrader, replica: ReplicaInfo) !void`
     - Stop replica (graceful shutdown)
     - Wait for confirmation
     - Signal to operator/orchestrator to replace binary
     - Wait for restart and health check
     - Wait for catch-up (replication_lag < threshold)
   - `pub fn checkHealth(self: *Upgrader) !HealthStatus`
     - Query /health/ready on all replicas
     - Query P99 latency from /metrics
     - Check error rate
   - `pub fn rollback(self: *Upgrader) !void`
     - Revert to previous version (signal to operator)
     - Log rollback reason

2. **Update cli.zig**: Add upgrade subcommand
   - `archerdb upgrade status --addresses=...` - Show current versions, identify primary
   - `archerdb upgrade start --addresses=... --target-version=... [--dry-run]`
   - `archerdb upgrade pause` - Pause in-progress upgrade
   - `archerdb upgrade resume` - Resume paused upgrade
   - `archerdb upgrade rollback` - Manual rollback trigger

Note: For Kubernetes, the CLI provides guidance and monitoring. The actual binary replacement happens via StatefulSet image update. The CLI orchestrates the process by checking health between pod updates.

Create comprehensive upgrade documentation:

1. **docs/upgrade-guide.md**: Complete upgrade guide

   **Overview section**:
   - Rolling upgrade philosophy (one node at a time)
   - Followers first, primary last
   - Health-based rollback triggers
   - Version compatibility (backwards compatible data format)

   **Pre-upgrade checklist**:
   - [ ] Review CHANGELOG for breaking changes
   - [ ] Verify backup is recent and verified
   - [ ] Check cluster health: all replicas healthy
   - [ ] Test new version in staging
   - [ ] Schedule maintenance window
   - [ ] Notify stakeholders

   **Version compatibility section**:
   - TigerBeetle model: each version specifies oldest compatible source
   - Sequential upgrades if too far behind (e.g., 1.0 -> 1.1 -> 1.2)
   - Data format backwards compatible (old versions can read new data)
   - Wire protocol: minor versions compatible, major may require simultaneous upgrade

   **Manual upgrade procedure** (bare metal):
   ```bash
   # 1. Identify primary
   archerdb upgrade status --addresses=node1:3000,node2:3000,node3:3000

   # 2. Upgrade followers (one at a time)
   ssh node2 "systemctl stop archerdb && cp /path/to/new/archerdb /usr/local/bin/ && systemctl start archerdb"
   # Wait for catch-up
   archerdb upgrade status --addresses=...

   # 3. Upgrade primary last
   ssh node1 "systemctl stop archerdb && cp /path/to/new/archerdb /usr/local/bin/ && systemctl start archerdb"
   ```

   **Kubernetes upgrade procedure**:
   ```bash
   # 1. Check current state
   kubectl get pods -l app=archerdb
   archerdb upgrade status --addresses=archerdb-0:3000,...

   # 2. Update image tag
   kubectl set image statefulset/archerdb archerdb=archerdb:v1.2.0

   # 3. Watch rollout (Kubernetes handles one-at-a-time)
   kubectl rollout status statefulset/archerdb

   # 4. Verify
   archerdb upgrade status --addresses=...
   ```

   **Health-based rollback triggers**:
   - Readiness probe fails for > 3 checks
   - P99 latency exceeds 2x baseline (configurable)
   - Error rate exceeds 1% (configurable)
   - Any replica fails to catch up within 5 minutes

   **Manual rollback procedure**:
   ```bash
   # Kubernetes
   kubectl rollout undo statefulset/archerdb

   # Bare metal
   ssh nodeX "systemctl stop archerdb && cp /usr/local/bin/archerdb.old /usr/local/bin/archerdb && systemctl start archerdb"
   ```

   **Post-upgrade verification**:
   - All replicas running new version
   - Cluster health normal
   - Replication lag minimal
   - Smoke test queries pass
   - Monitoring dashboards nominal

   **Troubleshooting section**:
   - Replica won't start after upgrade
   - Replication lag not decreasing
   - Primary won't failover
   - Wire protocol mismatch errors

   Cross-reference operations-runbook.md for general procedures.

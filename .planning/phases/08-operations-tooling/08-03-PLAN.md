2. **backup_coordinator.zig enhancements**:
   - Add replica role detection:
     ```zig
     const ReplicaRole = enum { primary, follower, unknown };
     fn getReplicaRole(self: *BackupCoordinator) ReplicaRole
     ```
   - Add follower-only check in backup trigger:
     ```zig
     pub fn shouldBackup(self: *BackupCoordinator) bool {
         if (self.config.options.follower_only) {
             const role = self.getReplicaRole();
             if (role == .primary) {
                 // Log that backup skipped on primary
                 return false;
             }
         }
         return self.scheduler.shouldRun(...);
     }
     ```
   - Add incremental backup tracking:
     - Track `last_backed_up_sequence: u64` in BackupState
     - Only upload blocks with sequence > last_backed_up_sequence
     - Update tracking after successful upload
   - Add backup progress callback:
     ```zig
     pub const ProgressCallback = *const fn(blocks_done: u64, blocks_total: u64) void;
     pub fn backupWithProgress(self: *BackupCoordinator, callback: ?ProgressCallback) !void
     ```

3. **Add metrics for backup monitoring** (if not already present):
   - `archerdb_backup_blocks_queued` (gauge)
   - `archerdb_backup_blocks_uploaded` (counter)
   - `archerdb_backup_last_success_timestamp` (gauge)
   - `archerdb_backup_role_skipped_total` (counter - for primary skips)

**CLI integration note**: The existing CLI uses BackupConfig to configure backup behavior. The follower_only option is set via:
- Config file: `backup.follower_only = true` in archerdb.conf

- The backup coordinator reads BackupConfig automatically; no new CLI flags are needed since configuration is handled through the standard config mechanism (same pattern as other backup options like backup.enabled, backup.provider, backup.bucket).

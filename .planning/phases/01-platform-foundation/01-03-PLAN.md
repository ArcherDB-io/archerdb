# Message Bus Error Handling

This document explains how the message bus classifies and handles errors.

## Error Classification

### Fatal Errors (Terminate Connection)

These errors indicate the connection is unusable:

| Error | Rationale |
|-------|-----------|
| Protocol violation | Peer sent invalid data, cannot trust further communication |
| Version mismatch | Incompatible protocol versions |
| Authentication failure | Peer failed to authenticate |
| Malformed message | Header/payload corruption |

**Action:** Terminate connection immediately with shutdown().

### Transient Errors (May Retry)

These errors may resolve on their own:

| Error | Rationale |
|-------|-----------|
| ConnectionResetByPeer | Peer closed connection (orderly or crash) |
| BrokenPipe | Write to closed connection |
| WouldBlock | Operation would block (timeout) |

**Action:** Log and terminate, but don't treat as system problem.

### Resource Exhaustion (Reject New Work)

These errors indicate system capacity limits:

| Error | Rationale |
|-------|-----------|
| SystemResources | OS resource limit (memory, buffers) |
| ProcessFdQuotaExceeded | Process file descriptor limit |
| SystemFdQuotaExceeded | System file descriptor limit |

**Action:** Log at WARN, reject new connections, keep existing connections alive.
Operator response: Increase limits or add capacity.

### Peer Eviction

When connection slots are exhausted and a replica needs to connect:

1. Prefer dropping client connections over unknown connections
2. Prefer dropping unknown connections over replica connections
3. Log at WARN level with evicted peer info
4. Emit metric (for alerting)
5. Emit cluster event (for operator automation)

## Platform Differences

### shutdown() Behavior

| Platform | Behavior |
|----------|----------|
| Linux | Pending I/O operations return immediately with error |
| Darwin | Similar, but timing may differ slightly |

Both platforms support graceful close via shutdown(.both) before close().

### Connection State Machine

```
free -> connecting -> connected -> terminating -> free
         |                              ^
         +------------------------------+
                  (on error)
```

## Configuration

Future work: Make timeout behavior configurable via:
- Connection timeout (how long to wait for connect)
- Idle timeout (how long inactive connection stays open)
- Retry policy (exponential backoff parameters)

Currently hardcoded in constants.zig.

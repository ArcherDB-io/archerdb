<objective>
Implement Flexible Paxos configuration to enable reduced quorum for lower latency commits.

Purpose: Classic Paxos requires majority quorum for all phases. Flexible Paxos (Howard et al., 2016) shows that only the intersection property matters: phase-1 quorum must intersect with phase-2 quorum. This allows reducing phase-2 quorum for lower commit latency while maintaining safety, at the cost of requiring larger phase-1 quorum during leader election.

Output: `src/vsr/flexible_paxos.zig` with quorum configuration and validation.
</objective>

Create Flexible Paxos quorum configuration following the academic paper's principles.

Key insight from Flexible Paxos: For safety, we only need Q1 (phase-1 quorum) and Q2 (phase-2 quorum) such that Q1 + Q2 > N. Classic Paxos uses Q1 = Q2 = ceil((N+1)/2), but we can reduce Q2 for faster commits.

Key structs:
```zig
pub const QuorumConfig = struct {
    cluster_size: u8,   // N: total replicas
    phase1_quorum: u8,  // Q1: leader election quorum
    phase2_quorum: u8,  // Q2: commit quorum

    // Presets
    pub const classic = QuorumPreset.classic;      // Q1 = Q2 = majority
    pub const fast_commit = QuorumPreset.fast_commit;  // Reduced Q2
    pub const strong_leader = QuorumPreset.strong_leader;  // Increased Q1

    pub fn validate(self: *const Self) !void { ... }
    pub fn majority(cluster_size: u8) u8 { ... }
};

pub const QuorumPreset = struct {
    // Classic: Q1 = Q2 = ceil((N+1)/2)
    pub fn classic(n: u8) QuorumConfig {
        const majority = (n / 2) + 1;
        return .{ .cluster_size = n, .phase1_quorum = majority, .phase2_quorum = majority };
    }

    // Fast commit: Q2 = ceil((N+1)/2) - 1, Q1 increased accordingly
    // Only valid for N >= 3
    pub fn fast_commit(n: u8) QuorumConfig {
        const majority = (n / 2) + 1;
        const q2 = if (majority > 1) majority - 1 else 1;
        const q1 = n - q2 + 1;  // Ensure Q1 + Q2 > N
        return .{ .cluster_size = n, .phase1_quorum = q1, .phase2_quorum = q2 };
    }

    // Strong leader: Q1 = N (unanimous), Q2 = 1 (single replica)
    // Maximum commit speed, but leader election requires all nodes
    pub fn strong_leader(n: u8) QuorumConfig {
        return .{ .cluster_size = n, .phase1_quorum = n, .phase2_quorum = 1 };
    }
};

pub const FlexiblePaxos = struct {
    config: QuorumConfig,

    pub fn hasPhase1Quorum(self: *const Self, votes: u8) bool {
        return votes >= self.config.phase1_quorum;
    }

    pub fn hasPhase2Quorum(self: *const Self, acks: u8) bool {
        return acks >= self.config.phase2_quorum;
    }
};

pub fn validateQuorums(config: QuorumConfig) !void {
    // Invariant: Q1 + Q2 > N
    if (config.phase1_quorum + config.phase2_quorum <= config.cluster_size) {
        return error.InvalidQuorumIntersection;
    }
    // Q1 and Q2 must be positive
    if (config.phase1_quorum == 0 or config.phase2_quorum == 0) {
        return error.InvalidQuorumZero;
    }
    // Q1 and Q2 cannot exceed N
    if (config.phase1_quorum > config.cluster_size or config.phase2_quorum > config.cluster_size) {
        return error.InvalidQuorumExceedsCluster;
    }
}
```

Document the tradeoffs clearly:
- Reducing Q2 speeds up commits but increases Q1, slowing leader election
- Strong leader is fast for commits but requires all nodes for election (availability risk)
- Classic is balanced and safest for most deployments

Create ServerConnectionPool following Java ConnectionPool patterns but adapted for Zig and server-side use.

Configuration (per CONTEXT.md decisions):
- `max_connections`: 32 default (range 16-64)
- `min_connections`: 4 default
- `idle_timeout_normal_ms`: 300_000 (5 min when idle)
- `idle_timeout_pressure_ms`: 30_000 (30s under memory pressure)
- `health_check_interval_ms`: 30_000 (30s per existing spec)
- `acquire_timeout_ms`: 5_000 (5s default)

Key structs:
```zig
pub const PoolConfig = struct {
    max_connections: u32 = 32,
    min_connections: u32 = 4,
    idle_timeout_normal_ms: u64 = 300_000,
    idle_timeout_pressure_ms: u64 = 30_000,
    health_check_interval_ms: u64 = 30_000,
    acquire_timeout_ms: u64 = 5_000,
};

pub const PooledConnection = struct {
    connection: *Connection,  // Underlying connection type
    pool: *ServerConnectionPool,
    last_used: i64,  // Timestamp
    in_use: bool,
};

pub const ServerConnectionPool = struct {
    config: PoolConfig,
    connections: []?PooledConnection,
    active_count: std.atomic.Value(u32),
    idle_count: std.atomic.Value(u32),
    metrics: *ClusterMetrics,

    pub fn acquire(self: *Self) !*PooledConnection
    pub fn release(self: *Self, conn: *PooledConnection) void
    pub fn isUnderMemoryPressure(self: *Self) bool
    fn reapIdleConnections(self: *Self) void
};
```

Memory pressure detection: Check system available memory (similar to RAM estimation in phase 13). If available < 20% of total, use shorter idle timeout.

Pool overflow handling (Claude's discretion): Queue waiters with timeout rather than fail-fast. Use bounded queue (max 64 waiters).
